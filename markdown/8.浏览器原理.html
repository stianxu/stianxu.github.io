浏览器 #

同源跨域 #

URL

  • URL 英文含义 “统一资源定位符”，中文译为“网址”，表示各种资源的互联网地址。
  • 协议：互联网支持多种协议，默认是 HTTP 协议。
  • 域名：指资源所在的网站名或服务器的名字，也可以使用 IP 地址作为域名使用。
  • 端口：不同端口提供不同服务，如果省略端口部分，将采用默认端口，HTTP 协议默认端口是 80。
  • 路径：表示服务器上文件的具体路径。
  • 参数：便于查询页面上不同的内容。
  • 锚点：网页内部的定位点，加载页面以后，会自动滚动到锚点所在的位置。

同源

  • 如果两个 URL 的协议、域名、端口都相同的话，那么这两个 URL 是同源。
  • 同源策略是 浏览器 的一种⽤于隔离潜在恶意⽂件的重要安全保护机制，服务器没有这个策略限制。

跨域

  • 当两个 URL 的协议、域名、端口三者之间任意一个不同即为跨域。

  • 项目上线后，访问地址与接口请求地址不一致，存在跨域。

  • 项目开发阶段，前后端联调接口，本地 localhost 与 请求接口地址 不一致存在跨域。

  • 由于浏览器同源策略的限制，当前端项目运行的服务地址和接口服务运行的地址出现 协议 域名 端口 三者有一个不一致就会出现跨域。跨域的两个条件：通过浏览器发起的 ajax 请求，协议 域名 端口 三者存在不同

代理服务器

  • 同源策略，是浏览器的安全策略，服务器与服务器之间，没有跨域问题。

  • 项目开发阶段，本地 localhost 与请求接口地址存在跨域，而渲染服务器和接口代理服务器不存在跨域，所以可以利用代理服务器转发请求。

  • 通过 webpack 配置 devServer 的 proxy 配置项来解决跨域。

  • js
    // webpack.config.js 部分配置
module.exports = {
  devServer: {
    // 代理配置
    proxy: {
      // 这里的api 表示如果我们的请求地址有/api的时候,就出触发代理机制
      '/api': {
        target: 'www.baidu.com', // 我们要代理请求的地址
        // 路径重写
        pathRewrite: {
          // 路径重写  localhost:8888/api/login  => www.baidu.com/api/login
          '^/api': ''
          // 假设我们想把 localhost:8888/api/login 变成www.baidu.com/login 就需要这么做 
        }
      },
    }
  }
}

CORS 跨源资源共享

  • 当项目上线后，部署访问的地址与接口请求地址存在跨域，这种一般都是需要服务端后端人员设置允许服务器除了自己以外的其它域名也可以访问加载这些资源。

JSONP

  • 利用 script 标签没有跨域限制来达到与第三方通信的目的，仅支持 GET，很少使用。
  • 通过 script 标签 src 属性，发送带有 callback 参数的 GET 请求，服务端将接口返回数据拼凑到 callback 函数中，返回给浏览器，浏览器解析执行，从而前端拿到 callback 函数返回的数据。

前端存储 #

cookie

  • 容量只有 4k，会为每个请求自动携带所有的 Cookies 数据，比较方便，但是浪费流量。

  • 每个 domain(站点) 限制存储 20 个 cookie。

  • API 比较原始，需要自行封装操作或使用第三方 js-cookie。

localStorage

  • 容量 5M，操作方便，永久性存储，除非手动删除。
  • localStorage.setItem('key', 'value')。
  • localStorage.getItem('key')。
  • localStorage.removeItem('key')。

sessionStorage

  • 功能基本与 localStorage 相似，但当前页面关闭后即被自动清理，不能在窗口之间共享。

Web SQL

  • 关系型数据库，2010 年被废弃。

IndexedDB

  • 非关系型数据库，用键值对进行存储，可进⾏快速读取操作，适合复杂 Web 存储场景，⽤ JS 操作⽅便，存储空间容量，大于等于 250MB，甚至没有上限。

同步异步 #

单线程

  • JS 是一门单线程执行的编程语言。同一时间只能做一件事情。
  • 如果前一个任务非常耗时，则后续的任务就不得不一直等待，从而导致程序假死的问题。
  • 为了防止某个耗时任务导致程序假死的问题，JS 把待执行的任务分为了两类：同步任务和异步任务

同步任务

  • 又叫做非耗时任务，指的是在主线程上排队执行的那些任务，只有前一个任务执行完毕，才能执行后一个任务。

异步任务

  • 又叫做耗时任务，异步任务由 JS 委托给宿主环境进行执行。

  • 当异步任务执行完成后，会通知 JS 主线程执行异步任务的回调函数。

  • JS 在处理异步任务时，利用的是事件循环机制：

    • 同步任务由 JS 主线程次序执行。
    • 异步任务委托给宿主环境执行。
    • 已完成的异步任务对应的回调函数，会被加入到任务队列中等待执行。
    • JS 主线程的执行栈被清空后，会读取任务队列中的回调函数，次序执行。
    • 这个过程是循环不断的，所以整个的这种运行机制又称为 EventLoop 事件循环。

  • 异步任务又分为 宏任务 和 微任务：

    • 由宿主环境发起的异步操作是宏任务
      • script 脚本的执行、setTimeout ，setInterval 一类的定时事件。
    • 由 JS 自身发起的异步操作是微任务
      • promise，async/await。
      • promise .then 之后才是异步的开始，async 函数只有从 await 往下才是异步的开始。

  • 宏任务 和 微任务的执行顺序：

    • 整体的 JS 代码这个宏任务先执行，同步代码执行完后有微任务执行微任务，没有微任务执行下一个宏任务，如此往复循环至结束。

缓存控制 #

浏览器缓存

  • 浏览器缓存主要是 HTTP 协议定义的缓存机制，分为强缓存 和 协商缓存。
  • 浏览器缓存减少了服务器的负担，提高了网站的性能，加快了客户端网页的加载速度，减少了多余网络数据传输。

强缓存

  • 浏览器先判断资源是否命中强缓存，如果命中则直接加载在缓存中的资源，并不会将请求发送到服务器。

协商缓存

  • 如果未命中强缓存，则浏览器会将资源加载请求发送到服务器，服务器来判断浏览器本地缓存是否失效，若资源仍可以使用，则服务器并不会返回资源信息，浏览器继续从缓存加载资源。

  • 如果未命中协商缓存，则服务器会将完整的资源返回给浏览器，浏览器加载新资源，并更新缓存。

触发操作

  • 地址栏回车：浏览器判断资源是否命中强缓存，未命中判断协商缓存，仍未命中服务器返回新资源。

  • F5 刷新：浏览器会将资源加载请求发送到服务器，服务器判断资源是否命中协商缓存，未命中返回新资源。

  • Ctrl+F5 强制刷新：申请最新资源，返回最新资源。

网页请求 #

当我们在浏览器的地址栏中输入：www.baidu.com，具体发生了什么

  • 对www.baidu.com这个网址进行 DNS 域名解析，得到对应的 IP 地址。
  • 根据这个 IP，找到对应的服务器，发起 TCP 的三次握手。
  • 建立 TCP 连接后，发起 HTTP 请求。
  • 服务器响应 HTTP 请求，浏览器得到 html 代码。
  • 浏览器解析 html 代码，请求 html 代码中的资源，如 js、css、图片等，渲染呈现给用户。
  • 服务过程完毕，关闭 TCP 连接，四次挥手。

界面渲染 #

界面渲染流程

  • 获取 HTML ⽂件并进⾏解析，生成一棵 DOM 树。

  • 解析 HTML 的同时也会解析 CSS，⽣成样式规则。

  • 根据 DOM 树和样式规则，生成一棵渲染树。

  • 进行重排，为每个节点分配⼀个显示的确切坐标位置。

  • 进⾏重绘，遍历渲染树节点，将元素呈现出来 。

浏览器是如何解析 CSS 选择器的

  • 在生成渲染树的过程中，渲染引擎会根据选择器提供的信息来遍历 DOM 树，找到对应的 DOM 节点后将样式规则附加到上面。
  • 浏览器遵循 “从右往左” 的规则来解析 CSS 选择器。

垃圾回收 #

垃圾回收

  • JS 代码运行时，需要分配内存空间来储存变量和值。当变量不在参与运行时，就需要系统收回被占用的内存空间，这就是垃圾回收。

  • 垃圾回收，核心思想就是如何判断内存是否已经不再会被使用了，如果是，就视为垃圾，释放掉。

引用计数法

  • IE 采用的引用计数算法，定义“内存不再使用”的标准就是看一个对象是否有指向它的引用，如果没有任何变量指向它了，说明该对象已经不再需要了。
  • 引用计数法存在一个致命的问题：循环引用。如果两个对象相互引用，尽管他们已不再使用，垃圾回收器不会进行回收，导致内存泄露。

标记清除法

  • 标记清除算法将“不再使用的对象”定义为“无法达到的对象”。从根部全局对象出发定时扫描内存中的对象。

  • 凡是能从根部到达的对象，都是还需要使用的。那些无法由根部出发触及到的对象被标记为不再使用，稍后进行回收。

内存泄露 #

内存泄漏

  • JS 中内存的分配和回收都是自动完成的，内存在不使用的时候会被垃圾回收器自动回收。如果不了解 JS 的内存管理机制，容易造成内存无法被回收的情况，导致内存泄漏。

内存的生命周期

  • 内存分配：当我们声明变量、函数、对象的时候，系统会自动为他们分配内存。

  • 内存使用：即读写内存，也就是使用变量、函数等。

  • 内存回收：使用完毕，由垃圾回收自动回收不再使用的内存。

  • 全局变量一般不会回收，一般局部变量的的值，不用了，会被自动回收掉。

哪些情况会导致内存泄漏

  • 意外的全局变量：由于使用未声明的变量，而意外的创建了一个全局变量，而使这个变量一直留在内存中无法被回收。
  • 被遗忘的计时器：设置了 setInterval 定时器，而忘记取消它，那么这个定时器会被一直留在内存中，而无法被回收。
  • 脱离 DOM 的引用：获取一个 DOM 元素的引用，而后面这个元素被删除，由于一直保留了对这个元素的引用，所以它也无法被回收。
  • 闭包：不合理的使用闭包，从而导致某些变量一直被留在内存当中。
